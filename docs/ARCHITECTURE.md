# 数学科インフラ：システムアーキテクチャ設計書 (v1.1)

## 1. コンセプト
- **1人での運用管理**: 複雑なツール（LDAPマネージャー等）を極力排除し、シンプルで自動化された構成を目指す。
- **ハイブリッド構成**: 停電に強い「学外（さくら）」と、速度に優れる「学内（オンプレ）」の役割分担。
- **Single Source of Truth**: ID管理のマスターを Windows Server 2025 AD に集約し、全システムへ同期・連携させる。

---

## 2. システム構成（コンテナ配置）

### 2.1 【学内：オンプレ集約サーバー】 (Lenovo SR250 V3)
学内LANの帯域を活かした重いデータの処理と、認証情報のマスター管理。
- **リバースプロキシ (Nginx Proxy Manager)**: SSL/アクセス管理。
- **Nextcloud (本体 + DB)**: 大容量ファイル共有。
- **LDAP / Samba AD (Windows Server 2025)**: **システム全体の認証マスター**。
- **学内ネットワーク管理**:
    - FreeRADIUS (Wi-Fi 802.1x 認証)
    - Kea / ISC DHCP (IP管理)
- **認証ゲートウェイ (Authentik)**: 二段階認証、セルフサービスによるパスワード変更、SSOの提供。

### 2.2 【学外：さくら専用サーバ / VPS】
可用性が最優先。停電時でも外部との連絡を維持する。
- **Mailcow (メールサーバスイート)**: SMTP, IMAP, Webmail。
- **ID同期スクリプト (Python自作)**: 学内ADの変更を Mailcow へ反映（LDAPマネージャーの代替）。

---

## 3. IDライフサイクルの変更（Legacy -> New）

### 3.1 現状 (Legacy)
- 権限の所在が分散（Linux、LDAPマネージャー、AD）。
- **LDAPマネージャー**: LinuxユーザーとAD（Proself用）の橋渡し、パスワード変更機能の提供。
- **課題**: ツールが複雑で、少人数での維持が困難。

### 3.2 移行後 (New)
- **マスター**: **AD (Windows Server 2025)** をマスターとする。
- **ユーザー作成**: AD上で1回だけ行う。
- **同期**: Pythonスクリプト (`ad-to-mailcow`) がAPI経由でメールアカウントを自動生成・更新。
- **パスワード変更**: Authentik または AD 連携のポータル（軽量な自作ページも検討可）が提供。
- **LDAPマネージャーの廃止**: 同期ロジックを透明なスクリプト（Python）に置き換えることで「ブラックボックス」を排除し、保守性を高める。

---

## 4. 検証戦略

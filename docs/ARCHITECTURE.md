# 数学科インフラ：システムアーキテクチャ設計書 (v1.1)

## 1. コンセプト
- **1人での運用管理**: 複雑なツール（LDAPマネージャー等）を極力排除し、シンプルで自動化された構成を目指す。
- **既存資産のコンテナ化**: DNS, DHCP, RADIUS, EFEREC は現在の設定ファイルを継承しつつ、Dockerコンテナへ移行・集約する。
- **オールオンプレミス起動 (Docker導入)**: 開発・初期運用フェーズでは、全てのコンテナを学内オンプレミスサーバー上で稼働させる。
- **将来のハイブリッド化**: 安定稼働後に必要に応じてMailcowコンテナ群をVPSへ移行可能にする。
- **Single Source of Truth**: ID管理のマスターを Windows Server 2025 AD に集約し、全コンテナへ同期・連携させる。

---

## 2. システム構成（コンテナ配置：フェーズ1）

### 2.1 【学内：オンプレ集約サーバー】 (Lenovo SR250 V3)
学内の全サービスを Docker 上に集約。既存設定を移行してコンテナ化する。
- **新規：認証/IDマスター (Windows Server 2025 AD)**: **システム全体の認証マスター**。
- **既存継承：DNS / DHCP (Bind9 / Kea)**: 既存設定をコンテナへ移植。
- **既存継承：RADIUS (FreeRADIUS)**: Wi-Fi / スイッチ認証。既存の `raddb` を流用。
- **既存継承：EFEREC**: コンテナ化して移行。
- **新規：リバースプロキシ (Nginx Proxy Manager)**: SSL/アクセス管理。
- **新規：Nextcloud (本体 + DB)**: 大容量ファイル共有。
- **新規：認証ゲートウェイ (Authentik)**: 二段階認証、ポータル機能。
- **新規：メールサーバスイート (Mailcow)**: **（暫定オンプレ稼働）** SMTP, IMAP, Webmail。
- **新規：ID同期スクリプト (Python自作)**: AD ↔ Mailcow 連携。

### 2.2 【将来の展望：学外拠点への切り出し】
安定稼働後、可用性（停電対策）のために特定の役割をVPSへ移行する。
- **Mailcowコンテナ群**: データを保持したままVPSへ移動（コマンド一発で移行可能）。
- **ID同期スクリプト**: 疎通先URL（API endpoint）をオンプレからVPSに変更するのみ。

---

## 3. IDライフサイクルの変更（Legacy -> New）

### 3.1 現状 (Legacy)
- 権限の所在が分散（Linux、LDAPマネージャー、AD）。
- **LDAPマネージャー**: LinuxユーザーとAD（Proself用）の橋渡し、パスワード変更機能の提供。
- **課題**: ツールが複雑で、少人数での維持が困難。

### 3.2 移行後 (New)
- **マスター**: **AD (Windows Server 2025)** をマスターとする。
- **ユーザー作成**: AD上で1回だけ行う。
- **同期**: Pythonスクリプト (`ad-to-mailcow`) がAPI経由でメールアカウントを自動生成・更新。
- **パスワード変更**: Authentik または AD 連携のポータル（軽量な自作ページも検討可）が提供。
- **LDAPマネージャーの廃止**: 同期ロジックを透明なスクリプト（Python）に置き換えることで「ブラックボックス」を排除し、保守性を高める。

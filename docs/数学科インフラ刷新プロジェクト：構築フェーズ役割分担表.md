# **数学科インフラ刷新プロジェクト：構築フェーズ役割分担・責任分界点定義書 (v2.2)**

## **1\. プロジェクト体制とロール（役割）の定義**

本プロジェクトでは、関係者の呼称と役割を以下のように明確に定義します。

* **【インフラ構築ベンダー】（現行保守業者様等を想定）**  
  * **役割:** ハードウェアの調達・設置、ネットワーク機器の設定、およびWindows OS・Active Directory（AD）の基礎構築を行う「土台作り」の専門家。  
  * **責任:** サーバーの電源が入り、ネットワークが物理的・論理的に繋がり、ADによる認証が正しく応答する状態を保証すること。  
* **【数学科システム管理者】（あなた / 内製開発担当）**  
  * **役割:** ベンダーが用意した土台（Linux OSやネットワーク）の上に、Dockerを用いてアプリケーション（メール、ファイル共有）を展開し、自動化プログラムを開発・実装する担当者。  
  * **責任:** メールやNextcloudなどのWebサービスが正常に動作し、ADと正しく同期・連携して安全に利用できる状態を保証すること。  
* **【大学本部ITセンター】**  
  * **役割:** 数学科のネットワークを大学全体のインフラ（Aruba Wi-Fi、外部インターネット）と繋ぐための設定変更を行う機関。

## **2\. 構築スコープと責任分界点（デマケーション）マトリクス**

「どこからどこまでが誰の作業か」を示す境界線（責任分界点）です。

| レイヤー | 構築対象システム・機器 | 構築担当 | 障害発生時の一次切り分け責任 |
| :---- | :---- | :---- | :---- |
| **物理層** | サーバー機器(ThinkSystem)、L3スイッチ、FW機器、LANケーブル | **インフラ構築ベンダー** | インフラ構築ベンダー |
| **OS・NW層** | Windows Server 2025、VLAN設定、ルーティング、FWルール | **インフラ構築ベンダー** | インフラ構築ベンダー |
| **認証基盤層** | Active Directory、NPS (RADIUS)、802.1X認証、MAB(MAC認証) | **インフラ構築ベンダー** | インフラ構築ベンダー |
| **全学接点** | Aruba Wi-Fi (CHUO-MATH) 転送設定、大学DNS・グローバルIP・(UPKI証明書) | **大学本部ITセンター** | 大学本部ITセンター |
| **仮想化層** | Linux OS (Ubuntu等)、Docker Engine、Docker Network | **数学科システム管理者** | 数学科システム管理者 |
| **アプリ層** | Nginx Proxy Manager, Authentik, Nextcloud, Mailcow | **数学科システム管理者** | 数学科システム管理者 |
| **ロジック層** | Python自動同期スクリプト (ad-to-mailcow) | **数学科システム管理者** | 数学科システム管理者 |

## **3\. 詳細タスクリスト（WBSベース）と用語解説**

各担当者が「具体的にどのような作業を行うか」の詳細です。

### **A. 【インフラ構築ベンダー】の作業タスク**

インフラ構築ベンダーには、以下の要件を満たす環境の納品を求めます。

1. **ハードウェア構築とOSインストール**  
   * ThinkSystem SR250 V3（2台）をラックへマウントし、ハードウェア障害に備えた\*\*RAID1（ミラーリング）\*\*を構築する。  
   * Windows Server 2025 Standard をインストールし、ライセンス認証を行う（OSは2コア構成での動作要件を満たすこと）。  
   * **Linux用環境の提供:** アプリを動かすためのLinux OSをインストールした状態（物理、またはWindows上のHyper-V仮想マシン）を提供する。  
2. **既存 Active Directory (AD) 基盤の流用・移行**  
   * 現行のActive Directoryドメインに新しいWindows Serverを追加のドメインコントローラーとして参加させ、役割の移行（FSMOロールの移行等）を安全に行う。  
   * 現在のユーザー情報（ID/パスワード）やグループ（OU構造）を完全に継承し、全ユーザーに対してパスワード再設定等の負担をかけない状態で新環境へ引き継ぐ。  
   * **用語解説【AD (Active Directory)】:** ネットワーク上の「誰が・どのパソコンから・何を使えるか」を一元管理するMicrosoftのデータベース機能。今回は「昔の電話帳の中身をそのまま新しい電話帳に引き継ぐ」形をとります。  
3. **NPS (RADIUS) による認証システムの構築**  
   * WindowsのNPS機能を用いて、後述のスイッチやWi-Fiからの認証要求（RADIUS）を受け付け、ADの情報を元に「接続許可/拒否」を返す設定を行う。  
   * **用語解説【RADIUS（ラディウス）】:** 「このIDとパスワードは正しいですか？」という問い合わせを受け付け、ADに確認を取りに行く通信のルール（プロトコル）。  
4. **ネットワーク機器（スイッチ・ファイアウォール）のコンフィグ作成**  
   * **802.1X認証の設定:** L3スイッチの各ポートに対して、PCを接続した際にOS画面でID/PWを要求する設定（EAP-PEAP等）を行う。  
   * **MAB (MAC認証) の設定:** プリンター等のMACアドレスをAD(NPS)に登録し、LANケーブルを挿すだけで特定のVLANへ参加できる例外設定を行う。  
   * **FW (Firepower) 設定:** インターネットから特定のポート（HTTP/HTTPS/SMTP/IMAP等）のみをLinuxサーバーへ通すルールを作成する。  
   * **用語解説【802.1X認証 / MAB】:** 802.1XはPCなどの「人」を証明書やパスワードで認証する強固な仕組み。MABはプリンターなど「画面がない機械」を、機器固有の番号（MACアドレス）で自動認識する仕組み。

### **B. 【数学科システム管理者】（内製開発）の作業タスク**

ベンダーから引き渡されたLinuxサーバー上で、以下の構築・開発を行います。

1. **Docker Compose によるコンテナ群の展開**  
   * 提供されたLinuxサーバーにDocker環境を構築し、設定ファイル（docker-compose.yml）を記述して各システムを立ち上げる。  
   * **Nginx Proxy Manager:** 外部からのWebアクセスを受け止め、裏側のアプリへ振り分ける。**Let's Encryptを用いたSSL証明書の自動更新を設定し、毎年の手動更新作業（UPKI証明書管理等）を廃止する。**（※大学のポリシー上UPKIが必須なドメインに限り、取得した証明書を手動でインポートする）  
   * **Authentik:** システムへのログイン時に、ADのパスワード認証に加え、スマホの認証アプリ等を用いた\*\*多要素認証（MFA）\*\*を強制するセキュリティゲートを作る。  
   * **Mailcow / Nextcloud:** メール送受信基盤と、ファイル共有基盤を構築する。  
2. **Python 自動同期スクリプトの開発・実装**  
   * ADサーバーへ定期的（例: 5分に1回）にLDAPプロトコルで通信し、新しく追加・変更されたユーザー情報を読み取るプログラム（Python）を作成する。  
   * 読み取った情報を、Mailcowの **REST API** を通じて送信し、メールアカウントの自動作成・パスワード同期を実現する。

### **C. 【大学本部ITセンター】への依頼タスク**

数学科と大学全体を繋ぐために必要な申請・設定作業です。

1. **Aruba Wi-Fi のRADIUS連携先変更**  
   * 数学科専用のWi-Fi（CHUO-MATH）に接続しようとしたユーザーの認証要求を、今回インフラ構築ベンダーが作成したWindows Server (NPS) へ転送するよう設定依頼を出す。  
2. **グローバルIPアドレス・DNSレコード・証明書の申請**  
   * mail.math.univ.ac.jp などの名前でアクセスできるよう、大学のDNSサーバーへレコード登録を依頼する。また、メールサーバー向けの逆引き（PTR）設定を強く依頼する。  
   * **（オプション）UPKI証明書の発行依頼:** もし大学のポリシー上、学外公開サーバーにUPKI証明書が必須と規定されている場合のみ、発行申請を行う。

## **4\. 障害発生時の切り分けルール（責任の明確化）**

トラブル発生時、どちらの責任範囲（調査担当）になるかを示す具体的なシナリオです。

* **ケース1：「研究室のLANケーブルを挿しても、認証画面が出ない、または繋がらない」**  
  * **担当:** インフラ構築ベンダー  
  * **理由:** 物理配線、L3スイッチの802.1X設定、またはWindows AD(NPS)のダウン等、インフラ土台の不具合であるため。  
* **ケース2：「特定のパソコンから、学科のプリンターで印刷ができない」**  
  * **担当:** インフラ構築ベンダー  
  * **理由:** L3スイッチ間のVLANルーティング、またはプリンターのMAC認証(MAB)が正しく動作していない可能性があるため。  
* **ケース3：「インターネットやWi-Fiは繋がるが、Nextcloudの画面が開かず、証明書エラーが出る」**  
  * **担当:** 数学科システム管理者  
  * **理由:** ネットワークインフラは正常に稼働しており、Linuxサーバー上のDockerコンテナ、またはNginx Proxy Managerの証明書自動更新設定に問題があるため。  
* **ケース4：「ADで新しい学生のIDを作ったのに、メールにログインできない」**  
  * **担当:** 数学科システム管理者  
  * **理由:** Pythonによる自動同期スクリプトのエラー、またはMailcowのAPI通信に問題が発生しているため。

## **5\. 検収条件（システムテスト・受け入れ基準）**

【インフラ構築ベンダー】の作業が完了し、数学科が「納品」を受け入れるための必須確認項目です。ベンダーには以下のテストシナリオを実機で実演・証明していただきます。

1. **有線LAN認証テスト:** 未設定のPCを情報コンセントに接続した際、通信が遮断されること。正しいADアカウント情報を入力した際のみ、インターネットへ通信可能になること。  
2. **無線LAN認証テスト:** スマホから「CHUO-MATH」へ接続し、ADアカウント情報で認証が成功し、通信可能になること。  
3. **プリンター接続テスト:** プリンターをLANに接続した際、MAC認証(MAB)によって自動的に通信が許可され、別VLANのPCからテスト印刷が成功すること。  
4. **AD連携通信テスト:** 数学科システム管理者が用意したLinuxサーバーから、ADサーバーのLDAPSポート(TCP 636)に対して、情報の読み取り（バインド）が成功すること。

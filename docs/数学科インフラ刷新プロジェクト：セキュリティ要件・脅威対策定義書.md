# **数学科インフラ刷新プロジェクト：セキュリティ・脅威対策定義書 (v1.1)**

## **1\. 本資料の目的**

本プロジェクトでは、システムを外部クラウドに委託せず、数学科内のサーバー（オンプレミス）で統合管理します。

本資料は、インフラ設計と運用において、現代の高度なサイバー脅威に対し「具体的にどのような技術・ツールを用いて防御するのか」を定義し、管理者の運用方針を明文化するものです。

## **2\. 脅威シナリオと具体的な技術対策**

### **A. 厳格化するメール送信要件への対応（スパム判定・不達リスク）**

2024年以降のGoogle/Yahoo等のバルク送信者ガイドライン強化に伴い、適切な送信ドメイン認証を行わないサーバーは即座にブラックリスト入りします。

* **実装レベルの対策（Mailcow \+ DNS）:**  
  1. **送信ドメイン認証の完備:** Mailcowの管理画面からDKIMキー（2048bit RSA等）を生成し、SPFレコード、DMARCポリシー（p=quarantine または p=reject）と共に大学ITセンターへDNS登録を依頼します。  
  2. **逆引き（PTR）設定の必須化:** 割り当てられたグローバルIPに対し、mail.math.univ.ac.jp への逆引き設定を大学ITセンターへ強く要求します（未設定の場合、Gmail等の受信拒否スコアが大幅に上がります）。  
  3. **List-Unsubscribe要件のクリア:** Mailcowの標準機能により、メーリングリスト等の送信時に「ワンクリック登録解除」のヘッダが自動付与されるため、Google側の要件を技術的に満たします。

**【非エンジニア向け解説】**

メールの「差出人偽造」を防ぐための電子署名（ハンコ）と、IPアドレスの「身元調査（逆引き）」を最新の規格に合わせて設定します。これがないと、数学科からの大事なメールが相手の迷惑メールフォルダに入ってしまいます。

### **B. Web脆弱性を突いた攻撃（SQLインジェクション・RCE等）**

Nginx Proxy Manager (NPM) が公開しているポート（80/443）を通じた、背後のアプリケーションへの攻撃です。

* **実装レベルの対策（OSS堅牢性 \+ WAF）:**  
  1. **アーキテクチャによる原理的防御:** NextcloudやMailcow内部では、データベース接続にORM（Object-Relational Mapping）やプリペアドステートメントが採用されており、SQLインジェクションが原理的に成立しにくい構造になっています。  
  2. **CrowdSecの導入（動的防御）:** NPMの背後に CrowdSec（Fail2banのモダン版）コンテナを配置します。NPMのアクセスログをCrowdSecがリアルタイム監視し、攻撃特有のアクセス（ディレクトリトラバーサルや悪意あるペイロード）を検知した瞬間に、そのIPアドレスからの通信をDockerレベル（またはiptables）で自動BAN（遮断）します。  
  3. **コンテナの爆速更新:** ゼロデイ脆弱性が発表された際は、docker compose pull && docker compose up \-d を実行するだけで、数分で安全なイメージへ差し替えが完了します。

**【非エンジニア向け解説】**

Webサイトへの攻撃に対し、アプリそのものの防弾チョッキ（堅牢な設計）に加え、不審者を自動で見つけて追い出すAI警備員（CrowdSec）を配置し、24時間自動で守らせます。

### **C. サプライチェーン攻撃（依存モジュールへのバックドア混入）**

NPMパッケージやPyPIモジュール等、開発に利用する外部ライブラリに悪意のあるコードが仕込まれる攻撃です。

* **実装レベルの対策（依存関係の完全な固定）:**  
  1. **Dockerイメージのダイジェスト指定:** docker-compose.yml において、latest タグの利用を避け、バージョン指定（例: v28.0.1）を行うか、より厳格にSHA256ダイジェスト（image@sha256:...）を指定し、イメージのすり替え・改ざんを防ぎます。  
  2. **Pythonスクリプト開発環境のロック:** ad-to-mailcow 同期スクリプトの開発において、pip ではなく uv や Poetry などのモダンなパッケージマネージャを利用します。uv.lock を生成して依存モジュールのハッシュ値を完全に固定し、意図しないバックドア入りアップデートが混入することを防ぎます。

**【非エンジニア向け解説】**

管理者が自作する自動化プログラムの「部品」に、悪意のある細工がされないよう、使う部品の「指紋（ハッシュ値）」を記録します。指紋が違う部品はシステムに組み込めないようにする、厳格な品質管理ルールです。

### **D. 認証情報の流出・総当たり攻撃（ブルートフォース）**

ADのパスワードが漏洩した場合や、外部からのリスト型攻撃による不正ログインです。

* **実装レベルの対策（ゼロトラストとMFAの強制）:**  
  1. **Authentikによる認証プロキシ:** 学外からのWebアクセス（Nextcloud等）は、必ずNPMからAuthentikへとリダイレクトされます。ここでADパスワードによる一次認証に加え、\*\*TOTP（スマホのワンタイムアプリ）や WebAuthn/FIDO2（指紋・顔認証・YubiKey等）による多要素認証（MFA）\*\*を必須とします。  
  2. **ネットワークアクセス制御 (NAC):** 学内インフラにおいては、L3スイッチとAD(NPS)を連携させた **802.1X認証（EAP-PEAP等）** を実装。物理的にLANケーブルを挿しても、ADで認証されない限りスイッチのポートが「Down」状態を維持し、IPアドレスすら割り当てられません。

**【非エンジニア向け解説】**

万が一、パスワードが盗まれても大丈夫な仕組みです。外からアクセスするには「本人のスマホや指紋」が必ず必要になり、学内でも「許可されていない怪しいPC」はLANケーブルを挿しても通信が完全に遮断されます。

## **3\. 管理者のセキュリティ運用義務**

システム管理者は、構築して終わりではなく、以下の運用（パッチマネジメントと監視）を継続する義務を負います。

1. **脆弱性情報のトラッキング:**  
   利用スタック（Docker, Nginx, Nextcloud, Mailcow, Authentik）のセキュリティリリース用メーリングリストやRSSを購読する。  
2. **定期アップデートサイクル:**  
   月1回の定期メンテナンスウィンドウを設け、各Dockerイメージの更新と、UbuntuホストOSのセキュリティパッチ（apt-get upgrade）を適用する。  
3. **バックアップとリカバリ訓練:**  
   Dockerのボリューム（永続データ）を定期的に別サーバー（または大学提供の外部ストレージ）へ暗号化して転送し、ランサムウェア等によるデータ暗号化被害を想定した「復元テスト」を半年に1回実施する。
